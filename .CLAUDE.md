## Project Architecture

This is a Vue 3 + TypeScript + Vite application with the following key architectural patterns:

### Application Structure

#### vue

Component Structure:

- Use single‑file components (.vue) with <script setup> and the Composition API.

- Keep components light and small; aim for one clear responsibility.

- Strongly type props and emits with TypeScript (defineProps, defineEmits).

- Name files and components in PascalCase (e.g. UserCard.vue).

- Prefix low‑level, reusable atoms with Base (e.g. BaseButton.vue).

- Export exactly one public component per file (private helpers can be declared with a leading \_).

Separation:

- Split UI into sub‑components by responsibility — avoid "god components".

- Move shared components to a global library (@/components/shared/).

- Keep private sub‑components inside the parent folder with a leading underscore (\_DropdownItem.vue).

- Keep styling concerns (<style scoped> or Tailwind) out of business logic.

State:

- Delegate complex state and logic to composables (useX) or Pinia stores.

- Declare computed properties and refs at the top of the setup() body.

- Use watchEffect/watch for side‑effects; avoid mixing logic and rendering.

- Prefer readonly props; mutate state via emits or stores.

- Return null (() => null) or an empty render if mandatory props are missing.

Style & Accessibility:

- Build mobile‑first and ensure responsive layouts.

- Prefer existing UI components (Tailwind, Headless UI, etc.) before rolling your own.

- Use semantic HTML (<button>, <nav>, etc.) and add appropriate ARIA attributes.

- Ensure keyboard navigation and visible focus indicators.

- Use Tailwind utility classes or <style scoped lang="scss">; avoid inline style="".

General:

- No Options API in new code.

- Avoid default exports for TypeScript modules (Vue SFC default exports are acceptable).

- Group imports: Vue, third‑party, internal aliases @/.

- Add unit tests (vitest, @testing-library/vue) for every public component.

## Architecture

### Core Technologies

- **Vue 3** with Composition API and `<script setup>` syntax
- **TypeScript** for type safety
- **Pinia** for state management
- **Vue Router** with hash history for extension compatibility
- **Tailwind CSS v4** for styling
- **GSAP** for animations
- **Axios** for API communication

### Browser Extension Structure

- **Manifest V3** with platform-specific builds
- **Content Scripts** inject the frontend on [monster-siren.hypergryph.com](http://monster-siren.hypergryph.com/)
- **Background Service Worker** handles extension lifecycle
- **Cross-platform compatibility** via prebuild scripts

### State Management (Pinia Stores)

- **usePlayQueueStore**: Music playback queue, shuffle/repeat modes, audio preloading
- **useFavourites**: User favorites with cross-platform storage (Chrome storage API/localStorage)
- **usePreferences**: User settings and preferences

### Key Components

- **Player**: Main audio player with preloading and resource validation
- **Playroom**: Full-screen player interface with lyrics and visualizations
- **ScrollingLyrics**: Animated lyrics display with auto-scroll and user interaction
- **PlayListItem/TrackItem**: Reusable music track components

### API Integration

- **Monster Siren API**: Fetches songs, albums, and metadata via `src/apis/index.ts`
- **Resource URL Validation**: Automatic refresh of cached URLs when servers rotate resources
- **Preloading System**: Smart audio preloading with cache management

### Browser Compatibility

- **Chrome**: Uses service worker, full CSP support
- **Firefox**: Uses background scripts, modified CSP, specific gecko settings
- **Safari**: Uses background page (background.html) instead of service worker
- **Prebuild Scripts**: Automatically modify manifest.json and HTML for each platform

### Storage Strategy

- **Favorites**: Stored in Chrome storage API (fallback to localStorage)
- **Preferences**: Browser-specific storage with graceful degradation
- **Audio Cache**: In-memory preloading with size limits

### Resource Management

- **Audio Preloading**: Validates and preloads next track during playback
- **URL Refresh Logic**: Checks resource availability before playback/preload
- **Cache Invalidation**: Automatic cleanup when resource URLs change

### Shuffle/Random Play Logic

- **shuffleList**: Array storing the shuffled order of original list indices
- **currentIndex**: In shuffle mode, index within shuffleList
- **Current song** in shuffle mode: `list[shuffleList[currentIndex]]`
- **getNextIndex**: Returns the actual list index of the next song to play

## Safari Extension Considerations

### Background Script Handling

Safari Web Extensions have different requirements for background scripts:

- **Background Page vs Service Worker**: Safari uses `background.page` instead of `service_worker`
- **Background HTML**: The prebuild script creates `background.html` that loads `background.js`
- **Manifest Configuration**: Uses `"background": { "page": "background.html", "persistent": false }`

### Auto-redirect Functionality

The auto-redirect feature in Safari may require special handling due to:

- Different WebKit extension APIs
- Safari's stricter security policies
- Tab management differences from Chromium

### Building for Safari

npm run build:safari   # Creates background.html and Safari-specific manifest

The Safari build process:

- Removes localhost development configurations
- Converts `service_worker` to `background.page`
- Creates `background.html` wrapper for `background.js`
- Adds Safari-specific browser settings

#### Typecript

Strict Types:
- Type everything explicitly
- Never use `any` or `unknown`
- Avoid `as` for type conversion
- Use type guards for assertions
- Use generics for reusable functions

Interfaces and Types:
- Use `interface` for extensible objects
- Use `type` for unions and primitives

Nullability:
- Avoid `null` and `undefined` in returns

Enumerations:
- Prefer string literal unions to enums
- Use const enums if needed
- Define enum values explicitly

Lint & Error:
- Catch errors as `unknown | Error`

Generics:
- Use descriptive type parameter names

#### State Management Pattern
- **Pinia stores** in `src/stores/` for global state
- **Composables** in `src/composable/` for reusable logic
- **Services** in `src/services/` organized by domain (entity, export, auth)
- Mock services available for development/testing

#### Component Architecture
- **Base components**: `src/components/base/` - reusable UI components
- **Layout system**: `src/layouts/` with multiple layout variants
- **Auto-import**: Components and composables are auto-imported via unplugin
